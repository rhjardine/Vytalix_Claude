# ADR-004 — Motores puros para lógica dental

## Estado

Accepted

## Fecha

2026-06

## Contexto

La lógica de cotización dental involucra cálculos financieros complejos: costos de materiales por procedimiento, márgenes dinámicos por complejidad clínica, ajuste geográfico de precios, conversión de divisas con tasa congelada e impacto de inventario. Colocar esta lógica en routers HTTP o en repositorios mezclaría responsabilidades, haría el código difícil de probar y crearía acoplamiento entre la lógica de negocio y la infraestructura de persistencia.

## Decisión

Toda la lógica de cálculo dental se implementa en motores puros sin efectos secundarios de I/O. Los motores reciben sus inputs como parámetros, producen outputs deterministas y no conocen la existencia de la base de datos, Redis, ni ningún servicio externo.

Los motores son:

- `DentalCostEngine` — calcula costo base por procedimiento (materiales, labor, overhead, factor geográfico)
- `MarginEngine` — calcula precio sugerido y utilidad neta a partir del costo
- `ExchangeEngine` — genera snapshots de tipo de cambio y convierte montos
- `InventoryEngine` — calcula impacto de inventario y registra movimientos sobre estado en memoria
- `QuoteOrchestrator` — coordina los cuatro motores anteriores para producir un `QuoteResult` completo con `TreatmentPlan` y `FinancialSnapshot` inmutable

Los routers (`dental-admin.router.ts`, `dental-commerce.router.ts`, `dental-core.router.ts`) invocan motores y repositorios; nunca contienen lógica de cálculo.

## Consecuencias

### Positivas

- Los motores son testeables unitariamente sin base de datos ni mocks de infraestructura.
- El mismo motor puede ser invocado desde un router HTTP, un test, o un script de migración sin modificación.
- El determinismo garantiza que el mismo `inputSnapshot` produce siempre el mismo resultado, requisito para reproducibilidad de auditoría clínica-financiera.
- `QuoteOrchestrator` como punto de entrada único simplifica el test de integración de la vertical.

### Negativas

- El `InventoryEngine` opera sobre estado en memoria en el Sprint actual; la sincronización con la base de datos requiere que el llamador persista el estado resultante. Esto crea una responsabilidad explícita en el router que puede omitirse inadvertidamente.
- Los motores de exchange usan tasas estáticas (`STATIC_FALLBACK_v1`); la integración con proveedores reales (BCV, Yadio) requiere un adaptador externo que no está implementado.

## Evidencia

- `src/dental/dental-cost.engine.ts` — clase `DentalCostEngine`; sin imports de DB ni HTTP; exporta `DENTAL_COST_ENGINE_VERSION = '1.0.0'`
- `src/dental/margin.engine.ts` — clase `MarginEngine`; calcula `suggestedPriceUsd = riskAdjustedCost / (1 - marginPct)`; sin side effects
- `src/dental/exchange.engine.ts` — clase `ExchangeEngine`; genera `ExchangeRateSnapshot` con TTL de 7 días; sin llamadas externas
- `src/dental/inventory.engine.ts` — clase `InventoryEngine`; opera sobre `InventoryState` pasado como argumento; retorna nuevo estado inmutable; detecta stock negativo con error 422
- `src/dental/quote.orchestrator.ts` — clase `QuoteOrchestrator`; coordina los cuatro motores; exporta `QUOTE_ORCHESTRATOR_VERSION = 'dental-quote-v2.0.0'`
- `src/dental/routers/dental-core.router.ts` — instancia motores como singletons; handlers invocan `quoteOrchestrator.generate()`; cero lógica de cálculo en el router
- `src/dental/routers/dental-commerce.router.ts` — handler de voucher invoca `dentalVoucherEngine.issue()`; resolución de precio delegada a `pricingRuleRepository.resolvePrice()`
- `docs/dental/CFE_DENTAL_ARCHITECTURE.md` — sección "Motores Puros": "No conectan a DB. No llaman APIs externas. Mismo input → mismo output (determinismo verificado en tests)"
