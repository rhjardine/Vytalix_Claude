# ADR-006 — Append-Only para registros clínicos y financieros

## Estado

Accepted

## Fecha

2026-06

## Contexto

Los registros de auditoría, snapshots financieros, movimientos de inventario y evaluaciones de edad biológica tienen valor legal y de trazabilidad. Si pueden ser modificados o eliminados retroactivamente, no pueden utilizarse como evidencia confiable de lo que ocurrió en un momento dado. La integridad temporal de estos registros es un requisito del dominio, no una preferencia de diseño.

## Decisión

Los registros de las tablas designadas son inmutables una vez escritos. La inmutabilidad se implementa en dos capas:

**Capa de base de datos:** Triggers `BEFORE UPDATE OR DELETE` ejecutan `prevent_modification()`, que lanza una excepción antes de permitir cualquier modificación. Esta protección opera independientemente del código de aplicación.

**Capa de dominio:** Los motores y repositorios solo exponen operaciones de inserción para estas entidades. No existen métodos `update` ni `delete` en sus interfaces.

Las tablas bajo este régimen son: `audit_logs`, `biological_age_assessments`, `billing_events`, `dental_audit_logs`, `dental_financial_snapshots`, `dental_inventory_movements`.

Los snapshots financieros (`FinancialSnapshot`) se crean con un `frozenAt` timestamp y un `algorithmVersion`, permitiendo reproducir el cálculo original incluso si los parámetros del motor cambian posteriormente.

## Consecuencias

### Positivas

- Un intento de `UPDATE` o `DELETE` sobre estas tablas falla con excepción de base de datos, incluso si el código de aplicación lo intenta por error.
- El historial de versiones de `TreatmentPlan` es permanente: cada modificación crea una nueva `TreatmentVersion`; las anteriores nunca se pierden.
- Los `InventoryMovement` forman un ledger auditado: el stock actual se deriva sumando todos los movimientos, no de un campo mutable.
- Los `dental_audit_logs` registran `before_state` y `after_state` como JSONB, permitiendo reconstruir el estado de cualquier entidad en cualquier punto del tiempo.

### Negativas

- Los errores de datos (ej. movimiento de inventario incorrecto) no pueden corregirse con `UPDATE`; requieren un movimiento compensatorio adicional, lo que aumenta el volumen de la tabla.
- Las evaluaciones de edad biológica erróneas solo pueden invalidarse mediante una nueva evaluación, no eliminando la incorrecta.
- El constraint CHECK `dal_event_type_check` en `dental_audit_logs` limita los tipos de evento aceptados; añadir un nuevo tipo requiere una migración SQL.

## Evidencia

- `prisma/migration_rls.sql` — función `prevent_modification()` con `RAISE EXCEPTION`; triggers `trg_immutable_assessments`, `trg_immutable_audit`, `trg_immutable_billing` sobre `biological_age_assessments`, `audit_logs`, `billing_events`
- `prisma/dental_sprint7_hardening.sql` — constraint `dal_event_type_check` en `dental_audit_logs` con enum de 8 eventos válidos; constraint `dfs_margin_bps_range` en `dental_financial_snapshots`
- `src/dental/services/audit.service.ts` — clase `AuditService` con método `record()`; solo `INSERT`; ningún método `update` ni `delete`; comentario: "append-only, never updated, never deleted"
- `src/dental/schemas/schema_hardened.prisma` — modelo `FinancialSnapshot` con campo `generatedAt @default(now())`; sin campo `updatedAt`; sin soft delete
- `src/dental/types.ts` — comentario en `FinancialSnapshot`: "Append-only. Una vez creada: NUNCA se modifica"
- `src/dental/inventory.engine.ts` — `recordMovement()` produce nuevo estado; el movimiento se agrega a `state.movements`; nunca se modifica un movimiento existente
- `docs/dental/CFE_DENTAL_DOMAIN_MODEL.md` — invariante 1: "Un `FinancialSnapshot` nunca se modifica — sólo se crea"; invariante 3: "Un `InventoryMovement` nunca se modifica — es append-only"
- `prisma/schema.prisma` — modelos `AuditLog` y `BillingEvent` sin campos `updatedAt` ni `deletedAt`
