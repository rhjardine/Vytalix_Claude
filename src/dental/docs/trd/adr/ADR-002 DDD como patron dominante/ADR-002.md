# ADR-002 — Aislamiento de dominios (Domain Isolation)

## Estado

Accepted

## Fecha

2026-06

## Contexto

El repositorio combina lógica clínica de alta criticidad, infraestructura multitenant, exposición HTTP, una vertical dental financiera y una superficie de presentación. Sin fronteras explícitas, cualquier componente podría importar de cualquier otro, haciendo imposible razonar sobre el impacto de un cambio o garantizar la integridad del sistema clínico.

## Decisión

El repositorio se organiza en cinco bounded contexts con fronteras de importación estrictamente unidireccionales:

**Core Clinical** (`src/core/`, `src/longevity/`): motor clínico aislado. No importa desde verticales satélite ni desde capas HTTP.

**Platform** (`src/platform/`, `src/api/middlewares/`, `src/api/pipelines/`): infraestructura multitenant compartida. Solo importa desde `src/shared/`.

**Commercial/API** (`src/api/handlers/`): adaptadores HTTP. Importa desde Core y desde Dental exclusivamente a través de `src/dental/index.ts`.

**Dental** (`src/dental/`): vertical satélite autónoma. No importa desde `src/core/` ni `src/longevity/` directamente. Su barrel `index.ts` es el único punto de entrada para importadores externos.

**Frontend** (`frontend-dental/`): superficie de presentación. No importa desde ningún módulo de servidor. Consume la API REST exclusivamente.

## Consecuencias

### Positivas

- Los sprints dentales (4–7) avanzan sin riesgo de romper el core clínico.
- La vertical dental puede probarse de forma aislada (281/281 tests en verde al cierre de Sprint 7).
- El frontend puede desplegarse independientemente del backend.
- Un defecto en un router HTTP no puede propagar lógica incorrecta al motor clínico.

### Negativas

- La comunicación entre Dental y Core requiere el event bus o contratos HTTP, añadiendo indirección.
- El barrel `src/dental/index.ts` debe mantenerse actualizado; un export faltante rompe integradores externos sin mensaje de error obvio.

## Evidencia

- `src/dental/index.ts` — barrel explícito con comentario: "Este es el ÚNICO archivo que código externo debería importar"
- `src/dental/types.ts` — encabezado: "Isolation contract: No imports from src/core, src/longevity, src/preventive"
- `src/server.ts` — monta `dentalAdminRouter`, `dentalCommerceRouter`, `dentalCoreRouter` importados desde `src/dental/index.ts` y routers individuales, no desde internals
- `frontend-dental/src/lib/api/client.ts` — contiene solo llamadas `fetch()` al backend; cero imports de módulos servidor
- `frontend-dental/src/types/dental.ts` — comentario: "SOLO tipos — sin lógica, sin imports del servidor"
- `frontend-dental/package.json` — dependencias: `next`, `react`, `react-dom`; ausencia total de `@prisma/client`, `pg`, `zod` o cualquier módulo de servidor
- `src/api/pipelines/pipeline-v2.orchestrator.ts` — comunicación Dental→Core mediante `eventBus.on('vitality.assessed', ...)`, nunca por import directo
- `docs/dental/CFE_DENTAL_ARCHITECTURE.md` — tabla de contratos de aislamiento con verificación: "`grep -r 'from.*dental' src/core` debe retornar vacío"
