# ADR-005 — OpenAPI como contrato único de integración

## Estado

Accepted

## Fecha

2026-06

## Contexto

El sistema tiene tres superficies de integración que deben mantenerse alineadas: los routers Express del backend, el cliente HTTP del frontend y los socios externos (Disglobal). Sin un artefacto canónico que las tres superficies referencien, cada una puede derivar en interpretaciones divergentes del contrato, generando errores silenciosos en producción.

## Decisión

Los archivos OpenAPI son el contrato único de integración. Definen rutas, métodos, schemas de request/response y códigos de estado. El orden de cambio es siempre: OpenAPI → implementación Express → consumidores. El orden inverso viola esta decisión.

El frontend no define contratos. `src/lib/api/client.ts` adapta sus llamadas y tipos al contrato OpenAPI del backend. Los tipos en `frontend-dental/src/types/dental.ts` son espejos explícitos, no definiciones originales.

Los socios externos (Disglobal) consumen los contratos OpenAPI publicados. La API Key autenticada en `requireApiKey()` aplica los scopes definidos en el contrato.

## Consecuencias

### Positivas

- Un socio externo puede validar su integración contra el OpenAPI sin necesitar acceso al código fuente.
- El cliente del frontend puede auditarse contra el OpenAPI para detectar campos faltantes o tipos incorrectos.
- Los scopes de API Key (`vitality:write`, `dental:read`, etc.) están definidos en el OpenAPI y aplicados en `api-key.middleware.ts`, creando trazabilidad directa entre contrato y enforcement.

### Negativas

- El campo `description` en `frontend-dental/package.json` muestra `cfe-dental-frontend` como nombre de paquete; la ruta base del cliente (`NEXT_PUBLIC_API_URL/v2/dental/...`) debe mantenerse sincronizada manualmente con los `paths` del OpenAPI dental.
- El archivo `dental-api-v2_synced.yaml` fue generado como variante sincronizada; la coexistencia de `dental-api-v2.yaml` y `dental-api-v2_synced.yaml` requiere disciplina para no mantener divergencia entre ambos.
- No existe en el repositorio un script de validación automática que compare el OpenAPI contra la implementación Express en cada commit.

## Evidencia

- `openapi/dental-api-v2_synced.yaml` — categorías de catálogo dental expandidas (`CONSULTATION`, `RESTORATION`, `ENDODONTICS`, etc.) sincronizadas con el vocabulario real de la implementación
- `src/dental/schemas/dental-schemas.ts` — `CreateCatalogItemSchema` usa el mismo enum de categorías que el OpenAPI
- `frontend-dental/src/lib/api/client.ts` — comentario de encabezado: "NO contiene lógica de negocio. TODO cálculo financiero delega al backend Express"
- `frontend-dental/src/types/dental.ts` — comentario: "Regla: si el backend cambia un tipo, este archivo debe actualizarse en paralelo"
- `src/api/middlewares/api-key.middleware.ts` — función `hasPermission()` aplica scopes definidos en el contrato OpenAPI
- `openapi/vytalix-platform-v2.yaml` — paths `/api/v2/vitality/assess`, `/api/v2/preventive/score`, etc. coinciden con los handlers en `src/api/handlers/external-v2.handler.ts`
- `src/server.ts` — log de startup imprime el manifiesto de rutas; todas las rutas listadas tienen correspondencia en los archivos OpenAPI
