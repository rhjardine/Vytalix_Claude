# ADR-007 — Dental como vertical autónoma dentro del monorepo

## Estado

Accepted

## Fecha

2026-06

## Contexto

La vertical CFE Dental tiene su propio ciclo de sprints, sus propios modelos de datos, sus propias reglas financieras y su propio frontend operativo. Coexiste con el core clínico en el mismo repositorio pero tiene una hoja de ruta, un conjunto de tablas y un equipo de trabajo distintos. Sin una separación explícita, los cambios de la vertical dental podrían afectar inadvertidamente al core clínico, y viceversa.

## Decisión

La vertical dental se implementa como satélite autónomo dentro del monorepo. `src/dental/` y `frontend-dental/` son las dos raíces de la vertical y operan con aislamiento de dominio estricto respecto al resto del repositorio.

`src/dental/` tiene su propio barrel de exportación (`index.ts`), sus propias tablas de base de datos (prefijo `dental_`), sus propias migraciones SQL, sus propios schemas Zod, sus propios servicios de auditoría y métricas Prometheus, y su propia documentación técnica (`src/dental/docs/trd/`).

`frontend-dental/` tiene su propio `package.json`, su propio `tsconfig.json`, su propio sistema de diseño (Tailwind configurado independientemente), y su propio ciclo de build (`next build`).

La comunicación de la vertical dental con el core ocurre exclusivamente a través del event bus o contratos HTTP. No mediante imports directos.

## Consecuencias

### Positivas

- Los Sprints 4–7 del backend dental se completaron (281/281 tests en verde) sin modificar ningún archivo del core clínico.
- El frontend dental (`frontend-dental/`) puede desplegarse y construirse independientemente: tiene su propio `next dev --port 3001`.
- Las migraciones SQL dentales (`20250902000000_dental_sprints4_5_tables`, `dental_sprint7_hardening.sql`) pueden aplicarse sin afectar el esquema del core.
- La observabilidad dental es independiente: `dentalMetrics` en `src/dental/services/prometheus-metrics.ts` tiene su propio registro Prometheus.

### Negativas

- La duplicación de algunos tipos entre `src/dental/types.ts` y `frontend-dental/src/types/dental.ts` requiere mantenimiento sincronizado manual.
- El `frontend-dental/package.json` usa Next.js 14.2.3, que tiene una vulnerabilidad de seguridad conocida (indicada en el `package-lock.json`); la actualización es independiente del core pero requiere acción.
- Las tablas con prefijo `dental_` conviven en la misma base de datos con las tablas del core; el aislamiento es lógico, no físico.

## Evidencia

- `frontend-dental/package.json` — nombre `cfe-dental-frontend`; scripts `dev --port 3001`, `build`, `typecheck`; sin dependencias del servidor
- `frontend-dental/tsconfig.json` — configuración TypeScript independiente con `"strict": true`
- `frontend-dental/tailwind.config.ts` — paleta de colores propia (`sidebar`, `accent`, `surface`, `ink`)
- `src/dental/index.ts` — barrel con comentario: "Este es el ÚNICO archivo que código externo debería importar. No importa desde src/core, src/longevity, src/preventive"
- `src/dental/docs/trd/adr/` — ocho carpetas ADR propias de la vertical (ADR-001 a ADR-008)
- `src/dental/services/prometheus-metrics.ts` — registro Prometheus propio con métricas `dental_quotes_created_total`, `dental_vouchers_issued_total`, etc.
- `src/dental/services/audit.service.ts` — servicio de auditoría propio, tabla `dental_audit_logs`, independiente de `src/platform/`
- `prisma/migrations/20250902000000_dental_sprints4_5_tables/migration.sql` — todas las tablas con prefijo `dental_`; migración independiente
- `src/server.ts` — monta routers dentales bajo `/api/v2/dental/*` con middleware `dentalTenantContext` propio
- `docs/dental/CFE_DENTAL_SPRINTS_CLOSURE.md` — reporte de cierre de sprints dentales independiente del core
