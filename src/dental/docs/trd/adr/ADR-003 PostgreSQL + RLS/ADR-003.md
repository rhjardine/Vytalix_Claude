# ADR-003 — Tenant Isolation mediante PostgreSQL RLS

## Estado

Accepted

## Fecha

2026-06

## Contexto

Vytalix es una plataforma multitenant. Múltiples clínicas operan sobre la misma base de datos. Un error de programación que omita el filtro de tenant en una consulta podría exponer datos de un tenant a otro, con consecuencias clínicas, legales y regulatorias graves. La validación solo a nivel de aplicación es insuficiente como única línea de defensa.

## Decisión

El aislamiento de tenant se implementa en dos capas obligatorias:

**Capa de aplicación:** Toda consulta a la base de datos pasa por `withTenant(tenantId, ...)`, que ejecuta `SELECT set_config('app.current_tenant_id', tenantId, true)` dentro de una transacción antes de cualquier operación de datos.

**Capa de base de datos:** PostgreSQL Row Level Security (RLS) está habilitado en todas las tablas con datos de tenant. Las políticas `tenant_isolation` filtran automáticamente filas cuyo `tenant_id` no coincide con `current_setting('app.current_tenant_id')`. Esta capa opera independientemente del código de aplicación.

El patrón `withTenant()` valida el formato UUID del `tenantId` antes de inyectarlo, previniendo inyección SQL.

## Consecuencias

### Positivas

- Un bug en el código de aplicación que omita un filtro `WHERE tenant_id = $1` no expone datos de otros tenants: RLS bloquea la query a nivel de motor de base de datos.
- El aislamiento es verificable mediante las políticas `pg_policies` sin ejecutar el código de aplicación.
- El middleware JWT (`tenant.middleware.ts`) valida que el `X-Tenant-ID` del header coincida con el `tenant_id` del token, añadiendo una tercera línea de defensa.

### Negativas

- `withTenant()` requiere que cada consulta se ejecute dentro de una transacción, incrementando la sobrecarga de conexión respecto a consultas directas.
- Las migraciones SQL deben incluir explícitamente `ENABLE ROW LEVEL SECURITY` y `CREATE POLICY` para cada tabla nueva; una tabla sin política pasa a ser invisible para todos los tenants (falla segura, pero puede generar confusión en desarrollo).
- El patrón `getDb().rawQuery()` directo, usado en operaciones de infraestructura (bootstrap, health check), bypasea `withTenant()` y está autorizado exclusivamente para esos contextos.

## Evidencia

- `prisma/migration_rls.sql` — habilita RLS en 21 tablas clínicas y de plataforma; crea políticas `tenant_isolation` mediante bloque `DO $$`
- `prisma/migrations/20250902000000_dental_sprints4_5_tables/migration.sql` — habilita RLS en 9 tablas dentales (`dental_audit_logs`, `dental_catalog_items`, `dental_pricing_rules`, `dental_exchange_rate_snapshots`, `dental_tenant_settings`, `dental_vouchers`, `dental_bookings`, `dental_financial_snapshots`, `dental_inventory_items`, `dental_inventory_movements`)
- `prisma/dental_sprint7_hardening.sql` — bloque `DO $$` que crea políticas `tenant_isolation` idempotentes para `dental_vouchers` y `dental_bookings` si no existen
- `src/platform/db.ts` — función `withTenant()` con validación UUID y `SET LOCAL app.current_tenant_id`
- `src/api/handlers/external-v2.handler.ts` — toda operación de datos usa `withTenant(tenantId, ...)`
- `src/dental/repositories/dental-financial.repositories.ts` — todas las queries usan `client` recibido de `withTenant()`
- `src/api/middlewares/tenant.middleware.ts` — valida consistencia entre JWT `tenant_id` y header `X-Tenant-ID`
- `src/api/middlewares/prisma.middleware.ts` — patrón alternativo Prisma: `getTenantDb()` usa `$transaction` con `set_config` antes de cada operación
